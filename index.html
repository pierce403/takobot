A highly autonomous octopus friend, imprinted to you.

tako-bot is built in Python with a docs-first memory system and deliberate Type 1 / Type 2 thinking, informed by modern productivity-hacking research. It can chat broadly, but only the operator can change identity, tools, permissions, and routines. It is web3-native via XMTP and Ethereum, with Farcaster support on the roadmap.

Python runtime Memory + Type 1/2 Operator imprint Web3 XMTP + Ethereum Farcaster roadmap

Quickstart

Bootstrap from your current directory (clone if needed), run first-wake SOUL prompts, then start Tako: 

curl -fsSL https://tako.bot/setup.sh | bash

# If you already have the repo:
./start.sh

# DM the printed "tako address" over XMTP
# Reply with: pair <code>
# Then: help / status / doctor

If installed, start.sh can optionally use one-shot local inference CLIs (codex/claude/gemini) to suggest SOUL defaults.

Developer utilities: ./tako.sh doctor and ./tako.sh hi <to> ["message"].

What happens on first run

  • Creates a Python virtual environment in .venv/ via uv.
  • Installs dependencies from requirements.txt via uv pip.
  • Installs the XMTP Python SDK (xmtp) via uv pip ‚Äî if it‚Äôs not available on PyPI, it clones xmtp-py and installs from source.
  • Generates .tako/keys.json with a wallet key and DB encryption key (unencrypted; file-perms only).
  • Creates the local XMTP database in .tako/xmtp-db/.
  • If available, optionally asks to use one-shot local inference CLIs (codex/claude/gemini) for SOUL identity prompt defaults.
  • Waits for the first inbound DM to initiate pairing; stores the operator imprint in .tako/operator.json.

Control plane

The operator XMTP channel is the only control plane. There is no user-facing configuration via environment variables or CLI flags.

  • Pairing is a challenge/response over XMTP: pair <code>.
  • After pairing, operator commands include: help, status, doctor, reimprint.
  • Runtime state lives under .tako/ and is never committed.

Note: treat .tako/keys.json as sensitive (and never commit anything under .tako/).

Operational notes

  • XMTP requires outbound network access; if messaging fails, double-check your HTTPS / HTTP2 egress.
  • The daemon retries XMTP stream subscriptions with backoff when transient group/identity stream errors occur.
  • Ensure uv is installed locally; bootstrap and dependency management run through it.
  • If uv is missing, start.sh attempts a user-local install automatically.
  • If you see DB errors (e.g., ‚Äúfile is not a database‚Äù), delete .tako/xmtp-db (or delete .tako/) to force a clean rebuild.
  • ENS resolution tries configured RPC endpoints first and may fall back to a public resolver if needed.
